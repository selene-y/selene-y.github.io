基本操作

        文件操作

  • touch file创建文件

  • less file 查看大文件，进入less编辑模式

  • vi file 编辑文件
    • esc(退出编辑模式), :q 退出

    • p 粘贴

常用命令

  • git init/git init dir初始化同时新建目录
  • git status
  • git add . 提交所有文件到暂存区
    • git add file 提交指定到暂存区
    • git add [dir] 提交目录到暂存区
  • `git rm [file1] [file2] … 删除工作区文件，并且将这次删除放入暂存区
  • git rm --cached [file] 停止追踪指定文件，但改文件会保留在工作区
  • `git mv [file-original] [file-renamed] … 改名文件，并且将这个改名放入暂存区
  • git commit 进入vi编辑模式，编写多行的修改日志
  • git commit -m "message" 提交到本地仓库，从而形成一次历史记录
  • git push
  • git log
  • git config --global user.email"XXX"
  • git config --global user.email "XXX"
    • 执行完以上两个命令之后，会在C: \users\administrator中生成文件.gitconfig保存用户信息，删除后需要重新配置用户信息
  • gitk打开图形界面
  • git remote --help打开使用帮助文档
  • git commit -a -m "message"将修改后的文件不经过暂存区，直接提交到本地仓库，形成一次历史记录

工作区、暂存区、本地仓库

  • 本地仓库通过git init出来的，存在于.git文件夹中
  • 暂存区的作用
    • 可以作为一个临时版本库
      • 可以将修改的文件分多次进行提交，使用git add filename可以单独提交文件到暂存区

GITHUB与GIT

  1. 在github上新建的项目新建项目
    • 使用git clone将远程的仓库下载到本地
      • 在github上创建仓库的时候会自动帮你初始化，所以git clone前不需要执行git init
      • 通过clone下来的远程仓库，git有一个远程仓库地址列表，git默认给远程仓库起一个别名，执行push的时候实际上就是把本地的版本提交到origin上
  2. 本地已有仓库
    • 如果是git init出来的仓库push的时候需要添加远程仓库地址，
      • git remote show查看所有的远程仓库信息
      • 添加远程仓库地址 git remote add 别名 address
    • 提交远程仓库git push 仓库地址别名 master
      • 使用git push --set-upstream 别名 master后续可以省略别名 master

撤销

  • 恢复暂存区的指定文件到工作区(文件已经被跟踪而且有历史记录)
    $ git checkout [file]
  • 恢复某个commit的指定文件到暂存区和工作区
    $ git check [commit] [file]

分支管理

基本操作

  • 列出所有的本地分支
    $ git branch
  • 基于当前分支创建一个新的分支，但是仍然处于这个分支
    $ git branch [branch-name]
  • 切换到指定分支，并切换工作区
    $ git checkout [branch-name]
  • 切换到上一个分支，交替和上一个分支进行切换
    $ git checkout -
  • 合并指定分支到当前分支
    $ git merge [branch]
  • 删除分支(不处于要删除的分支内)
    $ git branch -d [branch-name]

解决合并冲突

  • 在主干上修改好合并内容
    • 提交到暂存区
    • 提交本地仓库后
    • 再次进行分支合并
  • 切换到分支上，合并主干，将历史记录同步过来以及解决分支上的冲突文件

Github分支工作流

  1. 只有一个长期分支master，流程更简单
  2. 流程：
    • 从master中拉出新分支，不区分功能分支或不定分支
    • 新分支开发完成后，像master发送pullrequest
    • pull request既是一个通知让别人注意的的请求，又是一种对话机制，在对话过程中，你还可以不断提交新代码
    • 若pull request被接收，便可以合并进master，重新部署后，原来拉出的分支就会被删除
  3. github上的操作步骤
    • 点击fork将当前库复制到自己的账号下
    • clone到本地
    • 在clone下来的项目中拉出一个新的分支
      • 修改的时候最好是基于master拉出一个修改的分支
        • 在新分支上开发或者修改完成之后，提交到本地仓库，然后提交到自己仓库中fork霍老的仓库
    • 在github仓库首页，点击new pull request
